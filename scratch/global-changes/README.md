# Global Scope Changes for ServiceNow

This repository tracks **global-scope changes** to ServiceNow instances — the changes that
can't be CI/CD'd through the scoped-app pipeline. Each change lands as an update set XML
that a human developer reviews, imports, previews, and commits.

This is the **deliberate gating tier** of the agentic SDLC. Scoped apps get fully automated
deployment; global changes get human review of a deterministic artifact.

## Why a separate repo?

| Concern | Scoped apps (`servicenow-coder`) | Global changes (this repo) |
|---------|----------------------------------|----------------------------|
| Deploy mechanism | `now-sdk install` via CI/CD | Update set import via Studio |
| Approval flow | Tests pass → auto-merge | CAB review → human commit |
| Audit trail | Git history + ATF results | Git history + ServiceNow `sys_update_xml` |
| Blast radius | One scoped namespace | Cross-cutting platform behavior |
| Rollback | `now-sdk rollback <context>` | Update set back-out |

Mixing these would make the scoped-app repo's CI/CD harder to reason about. Keeping them
separate respects the different risk models.

## Repository structure

```
servicenow-global-changes/
├── changes/                          # One directory per change
│   ├── 2026-05-fsm-bridge/           # date prefix for ordering
│   │   ├── manifest.yaml             # change description + artifact list
│   │   ├── source/                   # source files (BR scripts, etc.)
│   │   │   ├── dispatch-br.js
│   │   │   └── ...
│   │   ├── built/                    # generated by build-update-set.js
│   │   │   ├── update-set.xml        # the importable artifact
│   │   │   ├── update-set.sha256     # integrity hash
│   │   │   ├── impact.md             # generated impact summary
│   │   │   └── preview.md            # generated preview
│   │   ├── README.md                 # change-specific runbook
│   │   └── .applied                  # marker file once committed on instance
│   └── ...
├── .github/
│   └── workflows/
│       └── validate.yml              # CI: re-run generator, ensure built/ is current
├── docs/
│   └── gating-procedure.md           # the standard review + commit procedure
└── README.md                         # this file
```

## How a change flows

```
1. Someone identifies a needed global change
                  │
                  ▼
2. AI generates manifest.yaml + source/ in this repo's
   changes/<date>-<name>/ directory
                  │
                  ▼
3. `node scripts/build-update-set.js changes/<date>-<name>/`
   (the generator lives in servicenow-coder/)
                  │
                  ▼
4. Generated `built/update-set.xml` + `impact.md` + `preview.md` committed
                  │
                  ▼
5. Pull request opened. Reviewer:
   ─ reads impact.md and preview.md
   ─ checks the source/ files
   ─ verifies update-set.sha256 matches the XML
   ─ optionally runs `python scripts/python/preview-update-set.py`
     against a non-prod target instance for live preview
                  │
                  ▼
6. PR approved.  CAB / change-management ticket created if required.
                  │
                  ▼
7. Developer downloads `built/update-set.xml`, imports via:
   ServiceNow → System Update Sets → Retrieved Update Sets
   → Import Update Set from XML
                  │
                  ▼
8. **Preview Update Set** in ServiceNow.  Reviewer compares against
   the generated preview.md.  Resolve any errors.
                  │
                  ▼
9. **Commit Update Set**.  Developer commits a `.applied` marker file
   to mark the change as deployed.
                  │
                  ▼
10. Post-import verification:
    `python scripts/python/preview-update-set.py
       changes/<date>-<name>/built/update-set.xml --post-import`
```

## Manifest format

See [manifest-schema.md](docs/manifest-schema.md) for the full reference (TODO).
Quick example:

```yaml
name: Add u_needs_field_service field to incident_task
description: |
  Required for the FSM bridge scoped app integration.
application: global
target_release: 2026-Q2

intent: |
  Enable analysts to flag an incident task as needing field service.

artifacts:
  - type: dictionary_entry
    table: incident_task
    element: u_needs_field_service
    column_label: Needs Field Service
    internal_type: boolean
    default_value: 'false'

  - type: business_rule
    table: incident_task
    name: FSM Bridge - Dispatch on Flag
    when: after
    action_insert: true
    action_update: true
    condition: current.u_needs_field_service == true
    script_file: source/dispatch-br.js

estimated_impact:
  tables_modified: [incident_task]
  rollback_complexity: low
```

## Supported artifact types

Phase 1 (today):
- `dictionary_entry` — `sys_dictionary` rows for custom fields on existing tables
- `business_rule` — `sys_script` rows for global-scope Business Rules

Phase 2 (planned):
- `db_object` — new tables (`sys_db_object`)
- `choice` — choice values (`sys_choice`)
- `ui_policy` — UI Policies (`sys_ui_policy`)
- `ui_action` — UI Actions (`sys_ui_action`)
- `script_include` — global-scope Script Includes

## Code Signing (enterprise extension)

If your environment has ServiceNow Vault with Code Signing enabled, the generated
update set XML can be cryptographically signed before being committed to the
protected instance. The Phase 1 sha256 hash provides tamper-evidence; Code Signing
provides cryptographic non-repudiation. See:

- `servicenow-docs/markdown/platform-security/sign-specific-records.md`
- `servicenow-docs/markdown/platform-security/code-signing-landing.md`

The signing step slots between Phase 1 step 4 (XML generated) and step 7 (XML imported).

## CI on this repo

The CI here doesn't deploy anything. It validates:

1. Every `changes/*/manifest.yaml` is valid YAML and matches the schema
2. `built/update-set.xml` is regenerable from `manifest.yaml` + `source/`
   (i.e., the committed XML matches what the generator would produce now)
3. `built/update-set.sha256` matches the actual file hash
4. No change has been merged without a `.applied` marker after 30 days
   (warning, not blocking)

If anyone hand-edits the XML, validation fails — forces all changes through the
generator.
